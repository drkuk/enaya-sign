# البنية التقنية وجاهزية النظام — إجابة مباشرة + خطة الإنتاج

## ١) إجابة أسئلتك الأربعة بوضوح

**ما تقنية الـ Backend؟**
بعد هذا التحديث: **Supabase** هو الـ Backend الفعلي (خادم مُدار + قاعدة بيانات + مصادقة + دوال خادمية). الواجهة (index.html / sign.html) تتحدث معه فقط، ولم تعد تحفظ العقود في المتصفح.

**ما قاعدة البيانات؟**
**PostgreSQL** (داخل Supabase)، بالجداول: `app_users` (المستخدمون وأدوارهم)، `clients`، `contracts`، `audit_log`. المخطط كامل في `schema.sql`.

**أين تُحفظ العقود والتوقيعات؟**
في جدول `contracts` على خادم Supabase — لا في المتصفح. التوقيع يُحفظ في العمود `signature` مربوطاً بالعقد عبر `code` الفريد. يبقى بعد إغلاق المتصفح وعلى أي جهاز.

**هل النظام جاهز للإنتاج؟**
الأساس التقني أصبح إنتاجياً (Backend + قاعدة + أدوار + أمان RLS). **لكنه يحتاج اختباراً فعلياً قبل عملاء حقيقيين** — راجع القسم ٣. لا تعتمده على عقود مُلزمة قبل إتمام تلك الخطوات.

## ٢) ما الذي تغيّر في هذا التحديث

- **تسجيل دخول إلزامي**: لا أحد يصل للوحة دون حساب. شاشة دخول عبر Supabase Auth.
- **نظام أدوار**: `admin` (أنت — صلاحية كاملة + إدارة المستخدمين) و `sender` (يُرسل عقوده فقط). كل مُرسِل يرى عقوده هو فقط؛ أنت ترى الكل.
- **إزالة Local Storage كمصدر**: العقود تُقرأ وتُكتب من قاعدة البيانات حصراً.
- **أمان RLS**: العميل لا يصل للجدول مباشرة، بل عبر دوال آمنة تُرجع عقداً واحداً فقط بالـ code. تعديل الرابط لا يكشف عقود الآخرين.
- **سجل عمليات**: إنشاء، فتح الرابط، توقيع — كلها تُسجَّل في `audit_log` بوقتها.
- **الرقم المرجعي** يُولَّد داخل القاعدة (دالة `next_ref`) لمنع التعارض.
- **التوقيع** يُحفظ في القاعدة عبر دالة `submit_signature` ولا يُفقد.

## ٣) ما يجب على مطوّر إنجازه قبل الإطلاق الفعلي (لا تتجاوزه)

سلّمتك أساساً صحيحاً، لكن هذه البنود تتطلب بيئة تطوير واختباراً تكرارياً — وهي ضرورية لعقود مُلزمة:

1. **اختبار شامل end-to-end**: إنشاء عقد بحساب مُرسِل، فتحه كعميل على جهاز آخر، التوقيع، ظهوره موقّعاً في اللوحة، والتأكد أن مُرسِلاً آخر لا يراه.
2. **اختبار اختراق RLS**: محاولة قراءة عقد بـ code آخر، ومحاولة وصول مُرسِل لعقود غيره عبر استدعاءات مباشرة — والتأكد أن القاعدة ترفض.
3. **البريد التلقائي**: نشر Edge Function + Resend (في `email-setup-plan.md`)، وتوثيق نطاق `enayah.org.sa` لتجنّب السبام، واختبار وصول الـ PDF فعلياً، وتسجيل `email_status`.
4. **الإشعارات**: نقل إشعار واتساب/الإيميل للمسؤول إلى جانب الخادم (Edge Function)، لا من صفحة العميل (لئلا يُكشف رقم المسؤول في كود العميل).
5. **النسخ الاحتياطي**: تفعيل Point-in-Time Recovery في Supabase، وسياسة استرجاع.
6. **انتهاء الصلاحية**: مهمة مجدولة (cron) تحوّل العقود منتهية المدة إلى `expired`.
7. **المراجعة القانونية**: حجية التوقيع الإلكتروني في السعودية، ونص الإقرار، وسجل التدقيق المقبول نظاماً — يراجعه مختص نظامي. أنا لست مرجعاً قانونياً.
8. **حدود حجم التوقيع**: صورة التوقيع (Data URL) قد تكبر؛ يُفضّل ضغطها أو رفعها لـ Supabase Storage بدل عمود نصّي عند التوسّع.

## ٤) تقدير صريح

- ما سُلّم: أساس Backend حقيقي قابل للبناء عليه — قاعدة بيانات، مصادقة، أدوار، أمان، سجل، وربط الواجهة.
- ما يتبقّى للإنتاج: اختبار وتأمين ونشر بريد ومراجعة قانونية — يحتاج مطوّراً ينفّذ ويختبر تكرارياً في بيئة فعلية.
- التوصية: استعن بمطوّر لإتمام القسم ٣. الملفات المسلّمة تختصر عليه أسابيع لأن التصميم والأمان والمخطط جاهزة.
